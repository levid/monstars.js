monstars.js

A Javascript MVC framework, using MooTools as the Core.

config.js

The config file describes the files you that make up your application. Here's an example config file:

init.core();
init.mvc('Storage', 'Model.Browser', 'View.Helpers');
init.app(function() {
	init.models('Task','Tag');
	init.controllers('Tasks', 'Tags');
	init.views('task/list', 'task/edit','tag/tag');
});

init.js

The init.js is the only file you need to directly specify in your HTML page. After passing it some paramaters after the query mark, it will load in the rest of your application for you.

To inform init.js about the rest of your application, you pass information as semi pseudo GET variables. Except they aren't variables. Simply, the first thing after the question mark is the path to your application's config.js file. Following a comma, you can also tell init.js if the mode is test/development, or production. Here's an example script tag:

<script src="/js/system/init.js?/js/app,development" type="text/javascript"></script>

Model

Models represent the actual data that an application will be using. It collects related information about a single object, and provides an easy and powerful API to access that information. Models also abstract away the specifics of where the data is stored. Whether you store the data on a server and need Ajax requests to fetch it, or you store it in the local SQLite database, the public API for Models stays the same, so the rest of your application can blissfully ignorant.

Model.Ajax

A Model.Ajax is one that finds its data on a server. Simply extend this, and all the related save and find methods will be making server requests. There is a static $urls property that you can set with url patterns if the default urls aren't sufficient.

Model.Browser

A Model.Browser stores its data locally in the browser, trying to use localStorage.

Model.SQL

Model.SQL keeps its data in a local SQLite database. Newer browsers have this available, as well as the Adobe AIR runtime.

Controller

Controllers handle all View events to make changes to the Model, and then re-render the view. Controllers either specify an element, or use the name of the Controller to own an element. The controller then observes events delegated from within this container.
